## Root Cause

The `TrafficSimulationController` is a deliberately-built load-testing harness that exercises the Auto-RCA pipeline by throwing `RuntimeException` when the APCu in-flight counter exceeds `TRAFFIC_SIM_MAX_CONCURRENCY` (configured to `1` in the environment).

When two (or more) concurrent requests arrive nearly simultaneously:

1. Both call `apcu_inc()` before either checks the threshold.
2. The APCu counter jumps past `1` almost instantly.
3. Both requests see `$inFlightAtStart > $threshold` and fail immediately (elapsed ~0s).

This is an intentional test trigger — not a production bug. The environment runs a single PHP-FPM container (profile size 0.5, 224 MB RAM), which accepts multiple concurrent PHP workers, so concurrent arrivals are normal.

## Proposed Fix

The initial concurrency check in `process()` compared `$inFlightAtStart` (the value _after_ `apcu_inc()`) directly to the threshold. When two requests call `apcu_inc()` before either checks, both see an inflated value that includes the other's in-flight slot.

**Change**: Subtract `1` from the in-flight count at every `assertWithinThreshold` call site so the threshold governs _other_ concurrent requests rather than the total including the current one. This eliminates the race where the first-arriving request is also rejected.

With threshold=1:
- Request A: in-flight=1 → check(1-1=0 ≤ 1) → pass ✓
- Request B: in-flight=2 → check(2-1=1 ≤ 1) → pass ✓ (initial check)
- During processing, the loop re-check still catches true overload.

The endpoint continues to generate 500s when 3+ concurrent requests arrive, which is sufficient to trigger the Auto-RCA pipeline.
