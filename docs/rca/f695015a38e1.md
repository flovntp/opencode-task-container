The `TrafficSimulationController` is a public load-testing endpoint specifically designed to exercise the Auto-RCA pipeline under stress. It intentionally throws a `RuntimeException` when the number of concurrent in-flight requests (tracked via APCu) exceeds a configurable threshold.

### How the exception is triggered

1. Two concurrent `GET /admin/auto-rca/simulate-traffic-exception` requests arrive nearly simultaneously.
2. Both atomically increment the APCu in-flight counter (`apcu_inc`) ‚Äî the first sees 1, the second sees 2.
3. Both enter `process()`, which immediately calls `assertWithinThreshold()`.
4. The first request (`inFlight=1 <= threshold=1`) passes and begins the CPU-intensive loop.
5. The second request (`inFlight=2 > threshold=1`) throws immediately ‚Äî `elapsed=0.00s` confirms zero processing time before failure.

## Proposed Fix

This is a **deliberate test exception** ‚Äî the controller works exactly as designed to generate 500s for the Auto-RCA pipeline. No code bug exists.

However, two observations worth a human review:

1. **Default threshold (`TRAFFIC_SIM_MAX_CONCURRENCY=1`)**: With a single instance and PHP-FPM's default `pm.max_children` (likely 5‚Äì10), any concurrent traffic spike above 1 request will immediately trigger the exception. If the intent is to test sustained overload rather than instant rejection, consider raising the default to match the PHP-FPM worker count.

2. **First check uses `$inFlightAtStart` instead of live APCu value**: `process()` receives the counter value captured at increment time. If a concurrent request finishes and decrements between the `apcu_inc` and the first `assertWithinThreshold` call, the first check could use a stale (higher) value. In practice this is harmless because the `finally` block runs after the throw, and the timing window is vanishingly small, but re-reading the live counter at the top of `process()` would make the first check consistent with the periodic loop checks.
